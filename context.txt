High-Level Summary
This repository is an EV opinion search engine project built around four major concerns:
1. Data collection from multiple platforms.
2. Dataset collation and preprocessing into one master corpus.
3. Search/indexing with Solr and a Flask UI.
4. Opinion classification with separate subjectivity and polarity models.

The repo contains both:
- A newer JSON-based pipeline centered on `data/output/master_corpus.json`, Solr core `ev_reviews`, and the Flask app in `app.py`.
- Older prototype scraping/indexing scripts that still refer to a different Solr core, `opinion_engine`.

Current Canonical Runtime Path
The most coherent current end-to-end path appears to be:
1. Collect platform JSON files into `data/*.json`.
2. Run `utils/collate_data.py` to normalize, deduplicate, and build `data/output/master_corpus.json`.
3. Optionally run `utils/extract_eval.py` to mark 1000 sampled rows as `eval` and export `data/output/eval_workspace.csv` for manual labeling.
4. Run `utils/ingest_to_solr.py` to wipe and load the master corpus into Solr core `ev_reviews`.
5. Start Solr via `docker compose up -d`.
6. Run `app.py` and use the Flask UI on port 8080.

Search App
Main file: `app.py`

What it does
- Runs a Flask app on `0.0.0.0:8080`.
- Connects to Solr core `http://localhost:8983/solr/ev_reviews`.
- Uses an embedded HTML template instead of separate templates/static assets.
- Supports:
  - keyword search
  - platform filtering through Solr faceting
  - result count and query time display
  - display of `source_target`, `platform`, `text`, `stars`, and `dataset_split`

Important implementation notes
- Empty search defaults to `*:*`.
- Solr query parser is `edismax`.
- Search field boosting is `text^2 source_target`.
- Results page size is fixed to 50.
- Sentiment filtering is only a UI placeholder and is not wired to model outputs yet.

Infrastructure

What it does
- Starts Solr 9.4 in Docker.
- Exposes port 8983.
- Precreates core `ev_reviews`.

Data Pipeline Details
Main file: `utils/collate_data.py`

What it does
- Scans `data/` for JSON files.
- Applies light microtext normalization:
  - expands some contractions
  - expands some EV-specific abbreviations (`ev`, `ice`)
  - expands `coe`
  - replaces a few emojis with words
  - removes URLs and `@mentions`
- Drops rows with fewer than 4 words after cleaning.
- Deduplicates by SHA-256 hash of `platform + lower(text)`.
- Writes standardized records to `data/output/master_corpus.json`.
- Prints corpus size and total word count, and warns if assignment thresholds are not met.

Important note
- `collate_data.py` expects all JSON input files directly under `data/`, not nested folders.

Evaluation Set Creation
Main file: `utils/extract_eval.py`

What it does
- Loads `data/output/master_corpus.json`.
- Samples up to 1000 rows with fixed seed 42.
- Uses TextBlob to generate pre-label guesses for:
  - subjectivity: `Opinionated` or `Neutral`
  - polarity: `Positive`, `Negative`, `Neutral`
- Marks sampled rows as `dataset_split = "eval"` and all others as `corpus`.
- Rewrites the master JSON in place.
- Exports a manual annotation CSV to `data/output/eval_workspace.csv`.

Important note
- This script mutates `master_corpus.json`, so running it changes the source used for Solr ingestion.

Solr Ingestion
Main file: `utils/ingest_to_solr.py`

What it does
- Reads `data/output/master_corpus.json`.
- Deletes all existing Solr documents with `*:*`.
- Adds the full dataset into Solr.

Important note
- This is a full replace, not incremental ingestion.

Scraping Layer
Primary folder: `scrapers/`

Status overview
- This folder mixes active scripts, one-off collection scripts, and older prototypes.
- Output format is inconsistent across scripts but generally resolves into platform JSON files later consumed by `collate_data.py`.

Most relevant scraper
- `scrapers/web_scraper.py`

What it currently does
- Uses Playwright plus BeautifulSoup.
- Actively orchestrates Team-BHP scraping in `main()`.
- Contains additional Trustpilot and other scraper functions, many commented out or disabled.
- Saves extracted data into `data/raw` according to the script, which does not match the current canonical `data/` input location used by `collate_data.py`.

Important scraper files
- `scrapers/web_scraper.py`: multipurpose web scraping script, only Team-BHP enabled in `main()`
- `scrapers/Bilibili_scraper.py`: interactive browser-based comment extraction to Excel, Chinese-language script, not wired into canonical JSON pipeline
- `scrapers/Translator.py`: translates a Bilibili Excel file into English, standalone utility
- `scrapers/reddit_scraper.py`: PRAW template with placeholder credentials, returns documents for old Solr-first design
- `scrapers/linkedin_scraper.py`: Selenium prototype, writes directly to old Solr core `opinion_engine`
- `scrapers/scraper.py`: old orchestrator for Reddit/X direct-to-Solr flow, also uses `opinion_engine`
- `scrapers/twitter/`: Twitter/X collection helpers and daily run artifacts

Key architectural conclusion
- The scraper layer is not fully unified.
- The repo currently has a stronger downstream pipeline than upstream ingestion discipline.
- Future cleanup should standardize all scrapers to emit the same JSON schema directly into `data/`.

Classification Layer
Primary folder: `classifiers/`

What it does
- Trains two classical ML text models:
  - subjectivity classifier
  - polarity classifier
- Uses TF-IDF + Logistic Regression for both.
- Stores trained models as `.joblib`.
- Provides prediction scripts for unlabeled text CSVs.
- Appends run summaries to `classifiers/report.md`.

Main training flow
- `classifiers/RunClassifiers.py`
  - orchestrates both model trainers
  - appends evaluation metrics to `classifiers/report.md`
- `classifiers/SubjectivityClassifier.py`
  - split: 70/30 stratified train/test
  - trains on `clean_text`
- `classifiers/PolarityClassifier.py`
  - split: 80/20 stratified train/test
  - filters to opinionated rows with valid polarity labels

Shared model definitions
- `classifiers/subjectivity_common.py`
- `classifiers/polarity_common.py`

Current modeling choice
- `TfidfVectorizer(ngram_range=(1, 2), max_features=10000)`
- `LogisticRegression(max_iter=1000, class_weight="balanced", random_state=42)`

Text preprocessing
- `classifiers/preprocessing.py`
- NLTK-based normalization:
  - contraction expansion
  - emoticon replacement
  - repeated-character normalization
  - URL/mention cleanup
  - hashtag term preservation
  - stopword removal with negations retained
  - lemmatization

Prediction flow
- `classifiers/RunPredictors.py`
  - runs subjectivity first
  - runs polarity only on rows predicted as opinionated
  - writes/updates `classifiers/predictions.csv`
  - appends run metadata to `classifiers/report.md`

Current classifier artifacts
- `classifiers/subjectivity_model.joblib`
- `classifiers/polarity_model.joblib`
- `classifiers/predictions.csv`
- `classifiers/report.md`
- `classifiers/scalability_results.csv`
- `classifiers/scalability_graph.png`

Observed latest recorded metrics from `classifiers/report.md`
- Subjectivity accuracy: 0.92
- Subjectivity ROC-AUC: 0.7052
- Polarity accuracy: 0.7043
- Polarity ROC-AUC: 0.7465

Important mismatch / bug candidate
- `PolarityClassifier.py` prepares `clean_text` in `prepare_opinionated_data()` but trains on `df["text"]`, not `df["clean_text"]`.
- `SubjectivityClassifier.py` does train on `clean_text`.
- This means the two training scripts are inconsistent in whether classifier preprocessing actually feeds the model.

Known Inconsistencies / Risks
1. Solr core mismatch:
   - Active app/infra uses `ev_reviews`
   - Older scrapers use `opinion_engine`

2. Input/output directory mismatch:
   - `collate_data.py` reads from `data/`
   - `web_scraper.py` writes to `data/raw`

3. Training inconsistency:
   - subjectivity model trains on preprocessed text
   - polarity model appears to train on raw text despite generating `clean_text`

4. Dependency reproducibility:
   - no single explicit dependency file was found
   - environment setup is currently implicit

5. Search/classification disconnect:
   - classifier outputs are not yet integrated back into Solr documents or Flask filters

6. Mixed artifact/source boundaries:
   - generated datasets, predictions, models, and reports live next to source code
   - future refactors should preserve provenance carefully

Recommended Mental Model For Future Work
- Treat this repo as three loosely coupled subsystems:
  1. corpus acquisition
  2. retrieval/search
  3. supervised text classification

- When changing something, first decide which subsystem is actually authoritative.
- Do not assume all scripts are part of the same current production path.
- Prefer the JSON master-corpus workflow unless explicitly reviving the older direct-to-Solr scrapers.

Safe Starting Points For Common Tasks
If the task is "run the search engine"
1. Start Docker Solr.
2. Confirm `ev_reviews` exists.
3. Rebuild corpus if needed with `utils/collate_data.py`.
4. Reingest with `utils/ingest_to_solr.py`.
5. Run `app.py`.

If the task is "update dataset"
1. Add or regenerate platform JSON files in `data/`.
2. Rebuild `master_corpus.json`.
3. Recreate eval split only if desired.
4. Reingest into Solr.

If the task is "work on ML"
1. Inspect `classifiers/eval.xls` or its successor labeled file.
2. Validate label columns and class balance.
3. Check the raw-vs-clean-text inconsistency in polarity training before making performance claims.
4. Retrain with `RunClassifiers.py`.
5. Use `RunPredictors.py` for unlabeled batch inference.
